# Torn Stocks Page — Verified DOM Facts

Structure observed on `https://www.torn.com/page.php?sid=stocks` via DevTools dumps supplied by the user. Use this file to confirm selectors **before** writing any DOM-interacting code — never guess beyond what is documented here.

Class names contain a random suffix generated by Torn's CSS-modules build. The suffix can change on redeploy. Always match on the **prefix** (e.g. `[class*="buy___"]`) rather than the full class name.

---

## Stock card

Each stock is a `<ul>` with a `stock___` class prefix, already mapped by `qtBuildMaps()` (`torn_stock_analyzer.js:2939`):

```js
$("ul[class^='stock_']").each(function() {
  var sym = $("img", $(this)).attr("src").split("logos/")[1].split(".svg")[0];
  qt_stockId[sym] = $(this).attr("id");        // Torn's internal stockId, used for postTrade
  qt_stocks[sym]  = $("div[class^='price_']", $(this));
  qt_stockRows[sym] = $(this);
});
```

So: `qt_stockRows[sym][0]` is the DOM `<ul>` for a given stock symbol.

## Buy / Sell trigger buttons (price row)

```html
<button type="button" class="torn-btn gray buy___fcM6a" i-data="i_230_456_43_34">buy</button>
<button type="button" class="torn-btn gray sell___eyGeI" i-data="i_622_456_45_34">sell</button>
```

- Visible text: lowercase `buy` / `sell`.
- Class prefixes: `buy___` and `sell___` (suffix rotates on rebuild).
- The `i-data` attribute encodes stock metadata (`i_<a>_<b>_<c>_<d>`) — **do not decode**, `qt_stockId` already gives us the internal id we need.
- Safe matchers:
  ```js
  cardEl.querySelector('[class*="buy___"]')
  cardEl.querySelector('[class*="sell___"]')
  ```

Clicking one of these expands the trade form (`manageBlock___`) inside the same stock card.

## Trade form (`manageBlock___`)

When buy or sell is clicked, the form expands inline within the stock's `<ul>`:

```html
<div class="manageBlock___PfiJh">
  <div class="message___ovnVh">
    <p>How many shares would you like to sell?</p>
    <p>Your <strong>2,000,000</strong> shares are worth<strong> $1,762,080,000</strong></p>
  </div>
  <div class="actions___PIYmF withInput___ZVcue">
    <div class="input-money-group success">
      <span title="..." class="input-money-symbol">
        $
        <input type="button" class="wai-btn" aria-label="...">  <!-- max-fill helper -->
      </span>
      <input data-testid="legacy-money-input" data-money="2000000" ... class="input-money">
      <input data-testid="legacy-money-input" data-money="2000000" ... class="input-money" type="hidden" value="2000000">
    </div>
    <button type="button" class="torn-btn gray sell___eyGeI" i-data="i_622_456_45_34">sell</button>
  </div>
  <div class="tt-money-input">  <!-- TornTools userscript widget; ignore -->
    <span>TornTools money input:</span>
    <input>
  </div>
</div>
```

### Form facts

- **Container:** `[class*="manageBlock___"]`. Mounts inside the stock's `<ul>` after the trigger click.
- **Inputs:** two `input[data-testid="legacy-money-input"]` elements — one visible, one `type="hidden"`. Set the visible one only; Torn's React mirrors it to the hidden one.
- **Input semantics:** despite the `class="input-money"` and `$` symbol, the value is **share count**, not dollars. The dump's `data-money="2000000"` matches the user's "2,000,000 shares" — not the "$1,762,080,000" total value.
- **TornTools injection:** the `tt-money-input` div is from another userscript. Do not target its `<input>`; the `data-testid="legacy-money-input"` selector excludes it.
- **Submit button:** same class as the price-row trigger (`[class*="buy___"]` for buy, `[class*="sell___"]` for sell), same `i-data`. May be the same DOM node moved into the form, or a duplicate — treat as "find the button by class+text on the card after the form is open."

### Two-step confirmation

After the input is filled and the form's submit button is clicked, the button text changes:

```
buy   →  Confirm Transaction
sell  →  Confirm Transaction
```

A second click on `Confirm Transaction` finalises the trade. The class stays the same; only the text changes.

Example "confirm" state observed:

```html
<button type="button" class="torn-btn gray buy___fcM6a" i-data="i_146_477_144_34">Confirm Transaction</button>
```

(`i-data` updates after the first click — irrelevant for our flow since we identify the button by class + the parent form, not by `i-data`.)

## Verified end-to-end flow

To execute a trade via Torn's own UI (matches StatForge's pattern of clicking native buttons rather than POSTing):

1. Find the stock card: `qt_stockRows[sym][0]` (already in TSA).
2. `cardEl.scrollIntoView({ behavior: "smooth", block: "center" })`.
3. Click the price-row trigger: `cardEl.querySelector('[class*="buy___"]')` or `[class*="sell___"]` whose visible text is `"buy"` / `"sell"`.
4. Wait for `[class*="manageBlock___"]` to appear inside the card (MutationObserver, ~1–2s timeout).
5. Set the visible `input[data-testid="legacy-money-input"]:not([type="hidden"])` to the **share count** (React-safe setter + `dispatchEvent("input", {bubbles:true})`).
6. Click the form's same-class button (text now `"buy"`/`"sell"`); poll for text change to `"Confirm Transaction"` (≤ ~750ms).
7. Click `"Confirm Transaction"`.
8. Observe success toast / DOM update; surface error toast if the button text never changes or returns to original.

This sequence:
- Makes zero non-API HTTP requests from the userscript.
- All requests to Torn's server are fired by Torn's own JS as a result of clicks on Torn's own buttons.
- Stays 1:1 with the user's click on TSA's Quick Trade button — one initiation, one trade.

## Buy form (verified)

Mirrors the sell form one-to-one. Same `manageBlock___` container, same `actions___` row, same `input-money-group`, same two `legacy-money-input` elements (visible + hidden mirror). Only differences:

- Form text: `"How many shares would you like to buy?"` instead of `"...sell?"`
- Helper line: `"$503,377 will buy you 571 shares"` (vs. `"Your X shares are worth $Y"`)
- Submit button class: `[class*="buy___"]` instead of `[class*="sell___"]`

Crucially, the **input value is the share count** (`data-money="571"` = 571 shares), not the dollar cost (`$503,377`). Same semantics as sell. The `$` symbol in `input-money-symbol` is purely cosmetic; it does not change what the input expects.

## Back button (verified — TornPDA, 2026-04-29)

After a successful trade Torn shows a "transaction complete" view with a back button that returns the form to the input state. Without dismissing it, the next trade-prep can't open a fresh form.

Identifiable by:

- **Text content:** `Back` (exact match, case-insensitive).
- **Class:** `torn-btn gray` — note the **absence** of a per-build `___xxxx` hash suffix that the `buy___`/`sell___` trigger buttons carry. The two-class form is the differentiator.
- **Scope:** lives inside `#stockmarketroot` but **outside** `[class*="manageBlock___"]` — a scoped `manageBlock___ button` query won't find it.

To dismiss programmatically, click via React fiber `onClick` (matches the rest of the trade flow's pattern), with native `.click()` as a fallback.
